## 场景设定

**背景**：你要做个"公司内部文档助手"

- 你们公司有 100 份内部技术文档（Word、PDF、Markdown）
- 新员工每次问"某某功能怎么配置"，老员工得翻半天文档
- 你想做个"智能客服"，问它就能直接给答案

**传统方案的问题**：
- 直接问 ChatGPT？→ 它没看过你公司文档，瞎答
- 关键词搜索？→ 搜"连接池"找不到"pool"，搜不准

**RAG 方案**：
> 让 AI "先读你的文档"，再根据文档回答。

## 第一阶段：摄入文档（一次性准备工作，离线完成）

### 场景
- 你有 3 份文档：`spring-boot-intro.txt`、`spring-boot-redis.txt`、`spring-boot-database.txt`
- 新员工小王稍后会问："**如何配置 Redis 连接池？**"

### 步骤 1：加载文档

```
读文件：
  data/documents/spring-boot-redis.txt
  ↓
得到一大段文字（2000 字）：
  "Spring Boot 集成 Redis 缓存

Redis 是一个高性能的内存键值存储系统..."
```

### 步骤 2：切片（Chunking）

**为什么切？** 因为一份文档 2000 字，但问题通常只和其中某一小段相关。切成小块，检索更精准。

```
原文档（2000 字）
  ↓ 按 chunk-size=500 切
chunk1（500字）："Spring Boot 集成 Redis 缓存\n\nRedis 是一个高性能的..."
chunk2（500字）："application.yml 配置：\nspring:\n  redis:\n    lettuce:\n      pool:\n        max-active: 8..."
chunk3（500字）："Redis Cluster 配置：\nspring:\n  redis:\n    cluster:\n      nodes:..."
chunk4（500字）："使用 RedisTemplate 操作 Redis..."
```

**类比**：把一本厚书拆成"章节"，方便快速翻到相关章节。

**关键参数**：
- `chunk-size`：每块多大（字符数）
- `chunk-overlap`：相邻块重叠多少字符（避免语义被切断）

### 步骤 3：向量化（Embedding）

**为什么要变成数字？** 因为计算机不懂"连接池"的意思，但会算数字的距离。

```
chunk2："application.yml 配置：spring redis lettuce pool max-active: 8..."
  ↓ 喂给 Embedding 模型（bge-small-zh-v15）
  ↓ 模型"读"这段话，提取语义特征
  ↓ 输出一串数字（512 个）
向量2：[0.26, -0.41, 0.75, 0.15, ..., -0.29]  ← 代表这段话的"语义指纹"
```

每个 chunk 都这样处理，得到 4 个向量。

**类比**：给每章书做"指纹识别"，标记这章讲什么主题。

**Embedding 模型的职责**：
- 输入：一段文字
- 输出：一串数字（向量）
- 特性：意思相近的文字 → 向量也相近

### 步骤 4：存进向量库

```
向量库（就是个特殊的数据库）
┌──────────┬────────────────────┬─────────────────────┐
│ chunk ID │ 原文（text）       │ 向量（embedding）    │
├──────────┼────────────────────┼─────────────────────┤
│ 1        │ "Spring Boot 集成..." │ [0.12, -0.34, ...]  │
│ 2        │ "application.yml..." │ [0.26, -0.41, ...]  │
│ 3        │ "Redis Cluster..."  │ [0.33, 0.67, ...]   │
│ 4        │ "使用 RedisTemplate"│ [-0.15, 0.89, ...]  │
└──────────┴────────────────────┴─────────────────────┘
```

**类比**：编一本"书籍索引"，每个主题都有对应的"指纹"。

---

**第一阶段完成！** 你的 3 份文档，变成了 12 个 chunk（每份约 4 个），每个 chunk 都有"语义指纹"（向量），存在向量库里。

这一步只做**一次**，以后不用重复（除非文档更新）。

## 第二阶段：回答问题（每次提问都要走这个流程）

小王问："**如何配置 Redis 连接池？**"

### 步骤 1：问题向量化

```
问题："如何配置 Redis 连接池？"
  ↓ 喂给同一个 Embedding 模型（bge-small-zh-v15）
  ↓ 模型"读"这个问题，提取语义
问题向量：[0.23, -0.45, 0.78, 0.12, ..., -0.34]  ← 问题的"语义指纹"
```

**关键**：问题和文档用**同一个 Embedding 模型**，这样它们的"指纹"才能比较。

**类比**：你拿着"连接池"这个关键词的"指纹"，去索引里找最像的章节。

### 步骤 2：检索相似片段（向量相似度计算）

```
拿问题向量，和向量库里的每个 chunk 向量比较：

问题向量 vs chunk1 向量  →  相似度 0.52  （不太像）
问题向量 vs chunk2 向量  →  相似度 0.83  （很像！）← 这个讲连接池配置
问题向量 vs chunk3 向量  →  相似度 0.76  （还行）  ← 这个讲集群
问题向量 vs chunk4 向量  →  相似度 0.45  （不像）
...
（对比所有 12 个 chunk）

按分数排序，取 top-3：
  1. chunk2（0.83）："application.yml 配置：spring redis lettuce pool..."
  2. chunk3（0.76）："Redis Cluster 配置..."
  3. chunk7（0.71）："lettuce.pool.max-active 参数说明..."
```

**怎么算"相似度"？** 用余弦相似度（一个数学公式），算两个向量的"夹角"：
- 夹角小（向量方向接近）→ 相似度高（0.8+）
- 夹角大（向量方向不同）→ 相似度低（0.3-）

**类比**：你拿着"指纹"在索引里找，找到 3 个最匹配的章节。

**关键参数**：
- `max-results`：召回几个片段（通常 3-5）
- `min-score`：最低相似度阈值（低于这个分数的片段直接丢弃）

### 步骤 3：拼接 Prompt（增强 Augmentation）

把检索到的 3 个 chunk + 问题 + 指令，拼成一大段话：

```
你是一个技术文档助手。请根据以下上下文信息回答用户的问题。

## 示例
问题：Spring Boot 的依赖注入和 @Autowired 注解有什么关系？
回答：依赖注入是机制，@Autowired 是应用...

---

上下文信息：
---
文档片段 1:
application.yml 配置：
spring:
  redis:
    lettuce:
      pool:
        max-active: 8
        max-idle: 8
        min-idle: 2
        max-wait: -1ms

文档片段 2:
Redis Cluster 配置：
spring:
  redis:
    cluster:
      nodes: ...

文档片段 3:
lettuce.pool.max-active 参数说明...
---

用户问题：如何配置 Redis 连接池？

回答：
```

**这就是发给 LLM 的完整 Prompt！**

**类比**：你把找到的 3 个章节复印下来，连同问题一起交给"专家"（LLM）。

**Prompt Engineering 的作用**：
- 定义角色："你是技术文档助手"
- 设定约束："只基于上下文回答"、"没有就说不知道"
- 提供示例：教 LLM 什么样的答案是好的
- 引导推理："可以综合多个片段"

### 步骤 4：LLM 生成答案（Generation）

```
把这一大段 Prompt 发给 Claude（LLM）
  ↓
Claude "读"这段材料（几秒钟）：
  - "哦，片段 1 讲的是连接池参数"
  - "片段 2 讲的是集群配置，和问题关系不大"
  - "片段 3 补充了参数说明"
  - "示例教我要识别'机制 vs 应用'"
  ↓
Claude 开始"写作"（递归预测下一个词）：
  "根据提供的文档，Redis 连接池配置在 application.yml 中...
   主要参数包括 max-active（最大连接数）、max-idle（最大空闲）...
   这些参数通过 spring.redis.lettuce.pool 配置..."
  ↓
生成完整答案（几百字）
```

**类比**：专家读完你给的章节，写一篇"读后感"回答你的问题。

**LLM 的能力**：
- 理解文本语义
- 跨片段综合信息
- 推理隐含关系
- 用自然语言组织答案

### 步骤 5：返回答案

```
答案通过 HTTP 接口返回给小王：
{
  "answer": "根据提供的文档，Redis 连接池配置在 application.yml 中...
             主要参数包括：
             - max-active: 最大连接数
             - max-idle: 最大空闲连接
             - min-idle: 最小空闲连接
             - max-wait: 最大等待时间
             这些参数通过 spring.redis.lettuce.pool 配置项设置。"
}
```

小王看到答案，问题解决！

## 关键角色分工

| 角色 | 干什么 | 类比 | 代码位置 |
|------|--------|------|---------|
| **DocumentService** | 加载文档、切片 | 图书管理员（整理书籍）| `DocumentService.java` |
| **Embedding 模型** | 文本 → 向量 | 指纹识别仪（给每章做指纹）| `bge-small-zh-v15` |
| **向量库** | 存向量、算相似度 | 书籍索引（快速查找）| `InMemoryEmbeddingStore` |
| **RagService** | 检索 + 拼 Prompt | 助理（找资料、整理给专家）| `RagService.java` |
| **LLM（Claude）** | 读材料、写答案 | 专家（读后写读后感）| `IflytekClaudeChatModel` |
| **RagController** | HTTP 接口 | 前台（接待用户）| `RagController.java` |

## 为什么 RAG 比直接问 ChatGPT 好？

### 场景 1：直接问 ChatGPT

```
你问："我们公司的 Redis 连接池怎么配置？"
  ↓
ChatGPT：
  "通常 Redis 连接池配置在 application.properties 中，
   参数包括 spring.redis.pool.max-active...
   （可能是旧版语法，可能不是你们公司的标准，可能编造）"
```

**问题**：
- ❌ ChatGPT 没看过你公司文档，凭训练数据瞎猜
- ❌ 可能给过时的配置
- ❌ 可能不符合你们公司规范
- ❌ 容易"幻觉"（编造看起来对但实际错的答案）

### 场景 2：用 RAG

```
你问："我们公司的 Redis 连接池怎么配置？"
  ↓
RAG 先去你公司文档里检索
  ↓ 找到"lettuce.pool 配置段落"
  ↓ 把这段话喂给 Claude
  ↓
Claude：
  "根据公司文档，Redis 连接池配置在 application.yml 中，
   使用 spring.redis.lettuce.pool 配置项...（准确引用你的文档）"
```

**优势**：
- ✅ 答案基于你的真实文档
- ✅ 不会编造（Prompt 约束"没有就说不知道"）
- ✅ 符合公司规范
- ✅ 文档更新了，答案也会更新（重新摄入即可）
- ✅ 可追溯（能看到答案基于哪些片段）

## 质量杠杆

通过实验验证的 4 个关键参数，直接影响 RAG 答案质量：

### 1. max-results（召回多少片段）

| 设置 | 效果 | 实验验证 |
|------|------|---------|
| 太少（1）| 容易漏掉正确片段 | 只召回集群配置，漏掉连接池配置 → 答案"我不知道" |
| 合适（3-5）| 给 LLM 足够材料，容错能力强 | 召回 3 个片段，正确的排第 2，LLM 自己挑出来答对 |
| 太多（10+）| LLM"读太多材料会走神"，成本高 | 未测试（通常不推荐）|

**推荐值**：3-5

### 2. Embedding 模型语言匹配

| 场景 | Embedding 模型 | 相似度分数 | 实验验证 |
|------|---------------|-----------|---------|
| 中文文档 + 中文问题 | 英文模型（all-MiniLM-L6-v2）| 0.68（低）| 中文对模型是"噪音"，靠蹭英文代码词勉强匹配 |
| 中文文档 + 中文问题 | 中文模型（bge-small-zh-v15）| 0.83（高）| 真正理解中文语义，分数提升 0.15 |

**核心结论**：Embedding 模型必须语言匹配，这是刚需不是优化。

### 3. chunk-size（切多大）

| 设置 | 效果 | 实验验证 |
|------|------|---------|
| 太小（200）| 把完整知识劈开，召回的是残片 | 连接池配置被切成两半，还混进不相关的"数据库连接池" |
| 合适（500）| 保持知识完整性 | 一段完整配置在一个 chunk 里，检索准确 |
| 太大（1000+）| 一块里混多个主题，检索精度下降 | 未测试（推测会降低精度）|

**核心结论**：chunk-size 要匹配"知识的自然颗粒度"（一段完整配置、一个完整概念）。

### 4. Prompt 风格

| 风格 | 设计 | 效果 | 实验验证（问"自动配置和连接池有什么关系"）|
|------|------|------|--------------------------------|
| 基础版 | 简单指令 + "没有就说不知道" | 诚实但无推理 | "我不知道...文档没说关系" |
| CoT（思维链）| "回答步骤 1、2、3、4" + "可以综合片段推理" | 主动综合分析 | 完整分析，推理出"互补关系" |
| Few-shot | 给一个类似问题的示例答案 | 照着示例的框架回答 | 简洁精准，结构清晰（机制/应用/总结）|

**核心结论**：
- 简单查询用基础版
- 复杂推理用 CoT
- 特定格式用 Few-shot

### 质量公式

```
RAG 质量 = (召回 top-N 兜底) × (Embedding 语言匹配) × 
           (chunk 知识完整性) × (Prompt 引导力) × (LLM 理解力)
```

这 5 个环节缺一不可，任何一个环节有问题都会影响最终答案。

## 核心概念速查表

### Embedding（嵌入/向量化）

- **是什么**：把文本变成数字向量的过程
- **为什么需要**：让计算机能"算"语义相似度
- **怎么工作**：相似的文本 → 相似的向量（通过训练学到的映射规律）
- **在 RAG 里**：检索阶段（把文档和问题都变成向量，算相似度）
- **关键点**：必须语言匹配（中文用中文模型）
- **例子**："Redis 连接池" → `[0.26, -0.41, 0.75, ...]`（512 个数字）

### LLM（大语言模型）

- **是什么**：Large Language Model，能理解和生成文本的 AI
- **为什么需要**：读材料、推理、组织答案
- **怎么工作**：递归预测"下一个词"（训练时学到了语言规律、知识、推理能力）
- **在 RAG 里**：生成阶段（读检索到的片段，写答案）
- **局限**：幻觉、知识过时、私有知识不知道 → RAG 解决
- **例子**：Claude Haiku、GPT-4、文心一言

### 向量库（Vector Store）

- **是什么**：专门存向量、算相似度的数据库
- **为什么需要**：快速找到"和问题最像的片段"
- **怎么工作**：用余弦相似度算向量之间的"夹角"
- **在 RAG 里**：桥接检索和生成（存第一阶段的向量，供第二阶段检索）
- **例子**：InMemoryEmbeddingStore（内存）、Qdrant（持久化）

### Chunk（切片/分块）

- **是什么**：把长文档切成小段
- **为什么需要**：检索更精准（问题通常只和某一小段相关）
- **怎么工作**：按字符数切（如 500 字一块），相邻块有重叠（避免语义被切断）
- **在 RAG 里**：摄入阶段（切完再向量化）
- **关键参数**：chunk-size（大小）、chunk-overlap（重叠）

### Prompt（提示词）

- **是什么**：发给 LLM 的那段话（指令 + 上下文 + 问题）
- **为什么需要**：引导 LLM 怎么回答（角色、约束、示例）
- **怎么优化**：Prompt Engineering（CoT、Few-shot、角色设定、约束条件）
- **在 RAG 里**：增强阶段（把检索到的片段拼进 Prompt）
- **例子**："你是技术文档助手。请根据以下上下文..."

## 一句话总结

> **RAG = 给 AI "开卷考试"的能力。**
> 
> 不再靠 AI"记忆"知识（容易记错、记过时），而是让它**现场翻书、引经据典**回答。
